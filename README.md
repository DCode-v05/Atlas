# Atlas — A2A Control Tower

**A real-time mission-control dashboard that shows 100 AI agents inside a simulated software company talking to each other — discovering colleagues, sharing only what's need-to-know, redacting or escalating sensitive data, and pulling a human in when a release is risky.**

## Overview

Atlas is a demonstrator of how AI agents communicate *like humans* inside an organisation — not how well they finish tasks, but how they talk. It builds a deterministic software company of exactly 100 agents across 12 departments (CEO → department heads → managers/leads → individual contributors), each with an A2A Agent Card, a clearance level, a standing goal, and private context only some colleagues should ever see. You task an agent with a prompt and watch the whole conversation play out on a control-tower screen: the agent gets routed, discovers who owns the context it needs, asks for it with a stated intent, and the owner decides whether to share, redact, deny, or escalate the request to a human. Sensitive disclosures get held back; a billing secret or an unreleased launch date lands in an approval queue you act on as the single operator.

The communication itself runs on a faithful, in-process implementation of the [A2A protocol](https://a2a-protocol.org), with an organisation-specific need-to-know layer bolted on through A2A's extension mechanism rather than by changing the core types. Every agent message, every routing choice, and every owner share-decision is generated by real Mistral models on Amazon Bedrock through the Converse API — there is no template fallback and no simulated mode. The focus throughout is the *quality of coordination*: how few hops it takes, how much got shared vs. redacted vs. withheld, how many redundant contacts were avoided, and how often a human had to step in.

I built Atlas during my AI Engineering internship at September AI, as one of several agentic prototypes exploring agent-to-agent coordination and need-to-know governance.

## Key Features

- **A 100-agent company, deterministically generated.** `generate_org(seed)` is a pure function — the same seed produces a byte-identical company. It builds a real hierarchy tree (every non-CEO has a resolvable `reports_to`), sets `clearance == level`, wires teams and three cross-cutting projects (`atlas-core`, `billing`, `mobile`), and seeds ~18 project secrets across every sensitivity tier onto sensible owners.
- **Live agent-to-agent communication graph.** A force-directed network of who is talking to whom — 1:1 vs. in groups — with edges colored by intent and outcome (shared / redacted / withheld / awaiting-approval).
- **Task an agent and watch it route.** Submit a prompt and see it pass the org-scope gate, get routed to the best-fit agent, open an A2A Task, discover the colleagues who own the context it needs, and request that context with a declared intent.
- **A two-layer need-to-know decision.** The data owner's own LLM call decides share / redact / deny / escalate, sitting underneath a deterministic, tighten-only policy engine that can only make a decision stricter, never looser — the codified compliance floor a real security function provides.
- **Human-in-the-loop approval queue.** Sensitive shares (payment secrets, MNPI, secret-tier items) escalate to you as the control-tower operator; you approve, redact, or deny, and the task resumes.
- **Out-of-scope gate.** Prompts unrelated to the company are blocked. Greetings and pleasantries are admitted (a bare "hi" routes to the CEO for a friendly reply); "what's the weather in Paris?" is gated.
- **Cron simulation.** Toggle a 15-second burst of autonomous agent activity that fires goals balanced across all departments, then auto-stops — or switch to a continuous mode that launches a goal roughly every 30 seconds until you turn it off.
- **Coordination-efficiency metrics.** Hops, messages, shared/redacted/denied counts, redundant contacts avoided, HITL escalations, distinct agents contacted, plus a compliance tile counting policy overrides and pre-gates.
- **Per-agent thinking and trace spans.** Every observable operation (route, judge-scope, judge-group, phrase, decide-share, policy-review) emits a trace span tagged `live=True` for a real Bedrock call or `live=False` for a deterministic step, so the UI can show exactly which decisions were the model's and which were the policy's.
- **An SSE-driven UI as the single contract.** The backend streams ~18 typed event types; the frontend mirrors the schema and warns on drift. No mock emitter — the UI develops against the real backend.

## How It Works

### Two layers, never conflated

Atlas keeps a hard line between the outside world and the agents:

- **External edge** (`atlas/api`, `atlas/events`) — the only real sockets. A REST API plus a Server-Sent-Events stream between the browser and the backend.
- **Internal bus** (`atlas/bus`) — agent-to-agent communication is in-process Python dispatch through a central **Router** that reproduces A2A method semantics (`message/send`, `tasks/get`, …). The Router is the single chokepoint where discovery, need-to-know, metrics, and event emission are enforced; agents cannot bypass it. 100 agents live in one process, not 100 servers.

Because the bus is in-process, the whole thing runs under a single uvicorn worker — that's deliberate, so the shared agent registry stays coherent. "100 agents running continuously" means a registry of 100 agent objects with a live status and heartbeat, woken event-driven by the Router or the cron simulator. Idle is an alive heartbeat with zero chatter, not 100 hot loops.

### The organisation

`generate_org(seed)` builds the hierarchy: 1 CEO, then Engineering (40 agents) plus Product, QA, DevOps, Sales, Design, Data, Marketing, Support, Security, and HR — 100 in total. Every agent gets a faithful A2A Agent Card with skills weighted by seniority (seniors carry leadership/liaison skills so *execution* prompts route to ICs while *strategy* prompts route up). Each agent has a standing `goal` (the agent analogue of a job description) and is paired 1:1 with a human `User` in a directory, so a prompt can be attributed to a specific person and the agent they operate.

### A2A fidelity and the extension mechanism

Core A2A types live in `atlas/a2a` (`AgentCard`, `Message`, `Part`, `Task` + `TaskState`, `Artifact`, `AgentExtension`). Organisation concepts attach through A2A's own extension URIs rather than polluting the core:

- `urn:atlas:ext:org-profile:v1` — dept / role / level / reportsTo / clearance on the card
- `urn:atlas:ext:need-to-know:v1` — sensitivity + scope on items; intent on messages
- `urn:atlas:ext:coordination:v1` — group and HITL signalling

### How a prompt flows

```
prompt → org-scope gate (LLM-judged) → Level-1 route (→ best agent) → open Task →
  agent identifies context needs → Level-2 discovery (→ owners) →
    for each owner: ask (with intent) → Policy pre-gate (deterministic floor):
        · floor already DENY/ESCALATE (denials, secrets) → decide outright, skip the owner LLM
        · else → owner LLM decides → Policy Engine reviews (tighten-only)
      → SHARE / REDACT / DENY / ESCALATE→HITL (operator approves) →
    or form a GROUP session when Mistral decides to coordinate the team →
  finalize Task → metrics emitted
```

The judgment calls are the model's. The **org-scope gate** is Mistral (lexical matching only as a fallback when the LLM is down). **Routing is fully LLM**: Mistral reads the whole directory — all 100 agent cards — and picks the owner; a deterministic skill-scorer survives only as a fallback. **Grouping is LLM-decided** too: Mistral chooses whether to coordinate as a group and which teammates to pull in, but only from the agent's real team roster, so it never invents people. What stays deterministic by design is *who owns what* and the team rosters — facts an LLM inventing private ownership would hallucinate.

### The need-to-know decision

This is the core of the project, and it's two layers.

**Layer 1 — the owner decides (LLM), under a policy pre-gate.** `llm.decide_share(requester, owner, item, intent)` returns SHARE / REDACT / DENY / ESCALATE from the model's own judgement, weighing sensitivity, the requester's role / clearance / teams / projects, and their stated reason. To save cost and latency, the deterministic floor (Layer 2) is computed *first*: when it already forces a DENY or ESCALATE — every denial, and every secret-tier item (which always needs four-eyes) — the owner's LLM is skipped, since the model can't loosen a deny anyway. The model is consulted only where its judgement can still change the result. If it's unreachable on that path, the request escalates to a human (`rule_id="LLM-UNAVAILABLE"`) rather than being decided in code.

**Layer 2 — the deterministic Policy Engine.** The owner's decision passes through `atlas/policy`, a tighten-only ABAC control that can make a decision stricter but never looser. It's ~11 single-action rules folded most-restrictive-wins over the lattice `SHARE < REDACT < ESCALATE < DENY`, each citing a named framework:

- **CLEARANCE-GATE** — no read-up (Bell–LaPadula; NIST 800-53 AC-3)
- **NEED-TO-KNOW** — out-of-scope confidential/restricted data downgraded to a summary (PCI Req 7 / NIST AC-6)
- **LEAST-PRIV-DENY** — restricted data deny-by-default absent a need (PCI Req 7 / AWS IAM)
- **PCI-SECRET** — live payment/API secrets escalate to four-eyes, or deny without a billing/incident nexus (PCI-DSS Req 3 & 7)
- **PII-PURPOSE / PII-MINIMISE** — no lawful basis for social use; minimum-necessary otherwise (GDPR Art. 5–6 / HIPAA)
- **HR-COMP** — compensation data restricted to People Ops + senior management (ISO 27001 A.5.12)
- **FINANCIAL-MNPI** — unreleased material non-public info needs human sign-off (SOX §404)
- **XDEPT-BOUNDARY** — restricted team/role data crossing departments is summarised (NIST AC-6)
- **SECRET-FOUR-EYES** — secret-tier disclosures always need an independent human approver (SoD; ISO 27001 A.5.3)
- **OFFICER-SELF-REVIEW** — the compliance authority can't self-approve its own sensitive data (NIST AC-5)

The same floor doubles as the pre-gate above. Reviews are attributed to the Security head, traced as `policy_review` spans, and metered as `policy_reviews` / `policy_overrides` / `policy_pregates`. The owner's *judgment* is the LLM's; the *floor* is the policy. (This replaced an earlier LLM "Policy Officer" agent with auditable, codified rules.)

### The LLM boundary

`BedrockProvider` (`atlas/llm`) is the only provider and drives both the user-prompt and cron paths via boto3's Bedrock Converse API, run in a worker thread since boto3 is synchronous. Two configurable Mistral model ids cover reasoning and phrasing (default `mistral.mistral-large-2402-v1:0`). Secret payloads are appended verbatim so the model can't silently drop them. To stay within Bedrock rate limits there's a per-model token bucket that agents *wait on* — `acquire()` blocks until a token is free, so calls are paced rather than dropped — plus a concurrency cap, SDK retries off, and a self-healing cooldown on `ThrottlingException`. The provider surfaces `throttled` and `errored` states over the `llm.status` event so the UI shows when conversations are degraded.

### The cron simulator

`atlas/cron` has two modes selected by `ATLAS_CRON_LOOP`: a 15-second burst (default) that fires a handful of autonomous goals then auto-stops, or a continuous mode that launches one goal roughly every 30 seconds until toggled off. Either way goals are round-robin balanced across all departments so it isn't Engineering-heavy, and each drives the exact same prompt pipeline from seeded agent-initiated tasks (the scope gate is skipped here, since goals are in-scope by construction).

### The frontend

A React 18 + Vite + TypeScript control-tower UI (`web/`) built around a five-tab center stage — Conversation, History, Network, Roster, Projects — with a teams panel, a right rail, and a metrics strip. State lives in a Zustand store fed by the SSE stream; the comms graph uses `react-force-graph-2d`. `atlas/events/schema.py` is the single source of truth for events, and `web/src/types.ts` mirrors it with the SSE client warning on drift.

## Results / Highlights

- **100 agents, 12 departments**, generated deterministically from a single seed (`ATLAS_SEED=42` by default) — same seed, identical company every time.
- **~18 seeded project secrets** spanning every sensitivity tier, attached to plausible owners across three cross-team projects.
- **~11 deterministic compliance rules**, each mapped to a named framework (NIST 800-53, PCI-DSS v4.0, GDPR, SOX, ISO 27001, Bell–LaPadula, XACML, AWS IAM).
- **68-test pytest suite** covering the org golden snapshot, the orchestrator HITL flow, the owner-LLM share decision plus every policy-engine rule, group need-to-know, LLM wiring and the payload guard, the Bedrock-credential requirement, cron, and API integration.
- **~18 typed SSE event types** forming the realtime contract between backend and frontend.
- Single-container build (~298 KB Python + ~120 KB TypeScript across the codebase) that serves API and UI on one port.

## Tech Stack

- **Languages:** Python 3.11+, TypeScript
- **Backend:** FastAPI, Pydantic v2 + pydantic-settings, `sse-starlette` (Server-Sent Events), uvicorn (single worker), httpx

## Getting Started

### Prerequisites

- Python 3.11+ and [`uv`](https://github.com/astral-sh/uv) (or Docker)
- Node 20+ for the frontend
- **Amazon Bedrock credentials with Mistral enabled.** Atlas runs real Mistral on every path and has no simulated mode — without credentials the app exits at startup with a clear error. On the AWS side you need: model access enabled (Bedrock → Model access → Mistral), the `bedrock:InvokeModel` permission, and an `AWS_REGION` where both Bedrock and that model are available.

### Installation

```bash
git clone https://github.com/DCode-v05/Atlas.git
cd Atlas

# credentials (.env is gitignored) — pick ONE auth option:
# A) classic IAM key + secret
printf 'AWS_ACCESS_KEY_ID=...\nAWS_SECRET_ACCESS_KEY=...\nAWS_REGION=us-east-1\n' > .env
# B) a Bedrock API key (bearer token) instead
printf 'AWS_BEARER_TOKEN_BEDROCK=...\nAWS_REGION=us-east-1\n' > .env
```

### Running

**Docker (single container — API + UI):**

```bash
docker compose up --build      # → http://localhost:8000
```

**Local dev:**

```bash
uv sync                                # backend deps into .venv
uv run python -m atlas                 # backend on :8000 (serves web/dist if built)

# in another shell, for live frontend dev with HMR:
cd web && npm install && npm run dev   # → http://localhost:5173 (proxies /api to :8000)
```

**Tests and typecheck:**

```bash
uv run pytest               # 68 tests; a fake LLM double runs them without a key
cd web && npm run typecheck # frontend type safety
```

## Usage

Open the UI and either task an agent or let the org run itself:

- **Task an agent.** Type a prompt. Try `"Fix the billing Stripe payment integration and get the API credentials"` (escalates to an approval), `"What is the Q3 launch date?"` (redacted), or `"What's the weather in Paris?"` (gated out of scope). Watch the agent route, discover owners, and request context with intent.
- **Hit SIMULATE.** Toggles the 15-second cron burst so the whole org lights up with autonomous coordination, then auto-stops.
- **Work the approval queue.** When a sensitive share escalates, approve / redact / deny it from the HITL queue — the task picks back up with your decision.
- **Read the metrics.** The strip tracks hops, messages, shared vs. redacted vs. withheld, redundant contacts avoided, and HITL escalations — the actual point of the project.

The backend is also usable directly: `POST /api/prompt`, `POST /api/cron`, the `POST /api/hitl/{id}/approve|deny` actions, `GET /api/org`, `GET /api/metrics`, and the `GET /api/events` SSE stream.
