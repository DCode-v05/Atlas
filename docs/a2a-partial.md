# A2A Compliance — Partial

Features that are **present but incomplete** — shaped differently from the spec, or defined in the code but never
exercised. Audited against **A2A v1.0.0**; see [`a2a.md`](./a2a.md) for the version-basis convention and the
Implemented / Not-implemented splits.

Each row is broken into three parts: **what's implemented (and why)**, **what's missing**, and **what completing
it would unlock** — the capability Atlas would gain by finishing the feature, framed against its actual
architecture (in-process bus, single worker, SSE edge, need-to-know + policy engine).

## 1. Agent Card and discovery

| Feature                             | Evidence                      | What's implemented (and why)                                                                                                                                                | What's missing                                                                                                                                      | What completing it would unlock                                                                                                                              |
| ----------------------------------- | ----------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `capabilities.streaming`          | `atlas/a2a/models.py:79-82` | The `streaming` bool is declared (true) on every card, so discovery advertises the intent in a spec-shaped field.                                                         | No A2A streaming backs the flag — there is no `SendStreamingMessage` / `SubscribeToTask` path.                                                 | Real per-task A2A SSE so external A2A clients (not just the browser) can stream a task's status/artifact updates, making the advertised flag truthful.       |
| Top-level `extensions` array      | `atlas/a2a/models.py:105`   | Atlas adds a top-level `extensions` list on the card and uses it to carry the org-profile extension, with a convenient accessor — one place to read an agent's identity. | It is a**non-spec** placement; the spec declares extensions only under `capabilities.extensions`.                                           | Moving org-profile under `capabilities.extensions` (or a real extended card) makes the card spec-valid for any external A2A consumer.                      |
| `AgentExtension` object           | `atlas/a2a/models.py:72-76` | `uri` + `version` + `required` + `metadata` cover declaration plus an arbitrary payload — enough for Atlas's three extensions; `required` now matches the spec.  | The spec's `description` and `params` fields (Atlas substitutes non-spec `version` / `metadata`).                                           | Self-documenting, introspectable extensions a generic A2A client could read and configure (`params`) without out-of-band knowledge.                        |
| `interfaces` (`AgentInterface`) | `atlas/a2a/models.py:90-93` | Each agent declares an interface with `transport=in-process`, `url=atlas://agent/{id}` — the honest description of the in-process bus.                                 | Spec naming (`supportedInterfaces`) and the per-interface `protocolBinding`, `protocolVersion`, `tenant` fields.                            | Declaring real bindings/versions would let Atlas expose agents over actual transports (HTTP+JSON / gRPC) and negotiate protocol versions.                    |
| Card signing (`signatures`)       | `atlas/a2a/models.py:106`   | An optional `signature` slot exists on the card.                                                                                                                          | The spec uses a**list** `signatures` of `AgentCardSignature` (a JWS over the JCS-canonicalised card); Atlas's is single and always empty. | Real JWS signing lets consumers verify a card is authentic and untampered — essential once cards leave the process.                                         |
| Card served over HTTP               | `atlas/api/routes.py:45-50` | `GET /api/agents/{id}/card` returns the card as a view-model, so the UI (and a curious client) can fetch it.                                                              | It is not the raw A2A card at the standard `/.well-known/agent-card.json` location.                                                               | Serving the raw card at the well-known URI makes Atlas agents discoverable by any standard A2A client.                                                       |

## 2. Core data objects

| Feature                 | Evidence                      | What's implemented (and why)                                                                    | What's missing                                                                                                                   | What completing it would unlock                                                                                                              |
| ----------------------- | ----------------------------- | ----------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `DataPart`            | `atlas/a2a/models.py:46-49` | The type is defined (`kind="data"`), so structured-data parts are representable in the model. | No live path produces one — messages are text-only.                                                                             | Machine-readable handoffs: agents could exchange structured payloads (a metrics blob, a policy-decision record) instead of prose.            |
| `FilePart`            | `atlas/a2a/models.py:52-55` | A `FilePart` type exists (a loose `file` dict), keeping the part-union shape complete.      | Never produced, and the shape matches neither 0.2.x FileWithBytes/Uri nor v1.0.0's `raw`/`url` + `filename`/`mediaType`. | Real file exchange — e.g. a Design agent attaching a mockup or a DevOps agent passing a log file by value or reference.                     |
| State:`canceled`      | `atlas/a2a/models.py:26`    | Defined and listed in `TERMINAL_STATES`, so the lifecycle vocabulary is complete.             | No cancel path ever sets it (no `CancelTask`).                                                                                 | An operator could abort a long/looping task or a runaway cron goal mid-flight, ending it cleanly instead of letting it run.                  |
| State:`rejected`      | `atlas/a2a/models.py:29`    | Defined as a terminal state.                                                                    | Never reached — nothing rejects a task.                                                                                         | An owner could `reject` a task it can't or won't take (out-of-scope, over-clearance), a cleaner outcome than `failed`.                   |
| State:`auth-required` | `atlas/a2a/models.py:28`    | Defined as an interrupted state.                                                                | Never used — there is no auth path.                                                                                             | With edge auth, a task could pause `auth-required` until the caller authenticates — the A2A-native way to gate on identity.               |
| `referenceTaskIds`    | `atlas/a2a/models.py:133`   | The field exists on `Message`, so a turn *can* cite related tasks.                          | It is never set.                                                                                                                 | Linking follow-ups to prior tasks gives the LLM explicit cross-task context ("re: the billing incident") and richer edges in the graph view. |

## 3. RPC methods

| Feature                          | Evidence                        | What's implemented (and why)                                                                                 | What's missing                                                                                                                                                  | What completing it would unlock                                                                                                       |
| -------------------------------- | ------------------------------- | ------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| Get Task (`GetTask`)           | `atlas/api/routes.py:103-109` | An equivalent REST read `GET /api/tasks/{id}` returns the full task, so the UI can inspect a task's state. | It is not the A2A method, not JSON-RPC, and ignores `historyLength`.                                                                                          | A spec-conformant `GetTask` (with history truncation) lets external A2A clients poll Atlas tasks.                                   |
| List Tasks (`ListTasks`)       | `atlas/api/routes.py:87-100`  | `GET /api/tasks` returns all tasks for the dashboard.                                                      | v1.0.0 makes List Tasks a core op with cursor pagination,`contextId`/`status` filters, `includeArtifacts`, and timestamp-desc ordering — Atlas has none. | Filtering/pagination that scales the task view and answers queries like "all working tasks in context X", powering richer monitoring. |
| `agent/card` (non-spec helper) | `atlas/a2a/methods.py`        | An Atlas-specific card read exists for the UI.                                                               | v1.0.0 has no such method — the card is fetched via the well-known URI or `GetExtendedAgentCard`.                                                            | Aligning to the standard discovery routes makes the helper unnecessary and the agent externally discoverable.                         |

| --- | --- | --- | --- | --- |
